## Problem Statement
Render UX has three foot-guns:
1. RenderButton stays enabled while a render is in progress — users click multiple times, get 409s
2. The 409 from routes/renders.ts:60 is silently swallowed by frontend — no toast, no visible feedback
3. If SSE drops during a long render, the client has no fallback — status appears frozen until refresh

## Root Cause / Motivation
- RenderButton.tsx:10-37 has zero status awareness; just opens a dialog on click
- Error code on conflict (renders.ts:60) is generic CONFLICT, not the documented RENDER_ALREADY_RUNNING
- No useRenderStatus hook exists; render polling/SSE is one-off per consumer

## Proposed Solution
1. Change error code at renders.ts:60 from CONFLICT to RENDER_ALREADY_RUNNING
2. Add useRenderStatus(projectId) hook — polls GET /renders/project/:projectId for active render
3. Pass activeRenderId to RenderButton; show Rendering… + spinner + disabled state while queued/running
4. On 409 response: toast "Render already in progress" + auto-switch UI to show the existing render
5. Add SSE → polling fallback in a new useRenderEvents(renderId) hook (only if SSE endpoint exists; otherwise stub the SSE branch and rely on polling)

## Alternatives Considered
- Disable button purely via local state: rejected — gets out of sync across tabs/refreshes
- WebSocket instead of SSE: out of scope; the existing pattern uses SSE for render events
- Inline polling in RenderButton: rejected — coupling. Extract to hook for reuse by RenderOptionsDialog too
